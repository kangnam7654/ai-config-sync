# Portfolio Review Checklist

포트폴리오 리뷰 시 참조하는 상세 체크리스트. career-consultant 에이전트가 GitHub, 개인 사이트, markdown 포트폴리오를 평가할 때 이 파일을 읽어 사용한다.

## 1. GitHub Portfolio 평가 기준

### Profile 평가

| 항목 | 기준 | 감점 조건 |
|------|------|-----------|
| Profile README | 존재하고 자기소개 포함 | README 없음 또는 기본 템플릿 그대로 |
| Pinned Repos | 3~6개 프로젝트 고정 | 고정 없음 또는 포크만 고정 |
| Bio | 역할 + 관심 분야 1줄 | 비어 있음 |
| 프로필 사진 | 전문적인 사진 또는 아바타 | 기본 아이콘 |
| 활동 그래프 | 최근 6개월 내 커밋 존재 | 1년 이상 비활동 |

### Repository 평가 (Pinned Repos 기준)

| 항목 | Good | Bad |
|------|------|-----|
| **README** | 프로젝트 설명, 스크린샷, 기술 스택, 설치/실행 방법 포함 | README 없음 또는 1줄짜리 |
| **프로젝트 설명** | 문제 정의 → 해결 방법 → 결과/임팩트 구조 | "~를 만들었습니다"만 기술 |
| **기술 스택** | 뱃지 또는 목록으로 명시, 선정 이유 간략 설명 | 언급 없음 |
| **데모** | 라이브 데모 링크 또는 GIF/스크린샷 | 코드만 존재 |
| **코드 품질** | 일관된 컨벤션, 린터 설정, 테스트 존재 | 커밋 메시지가 "fix", "update"만 반복 |
| **커밋 히스토리** | 의미 있는 커밋 메시지, 기능별 분리 | 1개 커밋에 모든 코드 |
| **라이선스** | LICENSE 파일 존재 | 없음 |

### README 구조 템플릿 (권장)

```markdown
# Project Name

> 한 줄 설명

## Problem
{어떤 문제를 해결하는가}

## Solution
{어떻게 해결했는가}

## Key Features
- Feature 1: {설명}
- Feature 2: {설명}

## Tech Stack
- Frontend: {기술 + 선정 이유}
- Backend: {기술 + 선정 이유}
- Database: {기술 + 선정 이유}

## Architecture
{Mermaid 다이어그램 또는 아키텍처 이미지}

## Demo
{라이브 링크 또는 GIF}

## Getting Started
{설치 및 실행 방법}

## What I Learned
{프로젝트를 통해 배운 점 — 주니어에게 특히 중요}
```

## 2. 프로젝트 설명 품질 평가

### 5가지 평가 기준

| # | 기준 | 평가 방법 |
|---|------|-----------|
| 1 | **문제 정의** | "왜 이 프로젝트를 만들었는가"가 명확히 기술되었는가 |
| 2 | **기술적 깊이** | 사용한 기술이 나열만 되었는가, 선택 이유와 트레이드오프가 설명되었는가 |
| 3 | **임팩트 수치화** | 성과가 정량적으로 표현되었는가 (사용자 수, 성능 개선, 비용 절감) |
| 4 | **역할 명확성** | 팀 프로젝트인 경우 본인의 기여 범위가 명확히 구분되었는가 |
| 5 | **학습 포인트** | 프로젝트를 통해 얻은 기술적 성장이 기술되었는가 |

### Before/After 예시

```
Before:
  "React와 Node.js로 쇼핑몰을 만들었습니다."

After:
  "소규모 의류 브랜드의 온라인 판매 채널 부재 문제를 해결하기 위해
   React + Node.js + PostgreSQL 기반 이커머스 플랫폼을 구축했다.
   Stripe 결제 연동, 재고 자동 관리, 주문 알림 기능을 구현하여
   런칭 2주 만에 150건의 주문을 처리했다.
   본인 기여: 백엔드 API 설계 및 결제 연동 전담 (팀 3인 중)."
```

## 3. 포트폴리오 유형별 추가 기준

### 개인 웹사이트 포트폴리오

| 항목 | 평가 기준 |
|------|-----------|
| 로딩 속도 | 3초 이내 로딩 |
| 모바일 반응형 | 모바일에서 깨지지 않는가 |
| 프로젝트 카드 | 썸네일 + 제목 + 한 줄 설명 + 기술 태그 |
| 상세 페이지 | 문제 → 해결 → 결과 구조 |
| 연락처 | 이메일, GitHub, LinkedIn 링크 |
| 404/에러 페이지 | 커스텀 에러 페이지 존재 (디테일 지표) |

### Markdown 포트폴리오 (PDF/문서형)

| 항목 | 평가 기준 |
|------|-----------|
| 목차 | 프로젝트 목록이 상단에 요약되어 있는가 |
| 프로젝트당 분량 | 0.5~1페이지 이내 |
| 스크린샷 | 각 프로젝트에 1개 이상의 시각 자료 |
| 링크 | GitHub repo, 라이브 데모 URL 포함 |
| 날짜 | 프로젝트 기간 명시 (최신순 정렬) |

## 4. 시니어리티별 포트폴리오 전략

| 시니어리티 | 프로젝트 수 | 강조 포인트 |
|-----------|------------|------------|
| 주니어 (0~2년) | 3~5개 | 학습 능력, 기술 다양성, 완성도 |
| 미들 (3~5년) | 3~4개 | 기술적 깊이, 문제 해결 과정, 팀 협업 |
| 시니어 (5~10년) | 2~3개 | 아키텍처 결정, 임팩트 규모, 리더십 |
| 리드/스태프 (10년+) | 1~2개 | 조직 레벨 임팩트, 기술 전략, 멘토링 |
