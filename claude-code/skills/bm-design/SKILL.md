# BM Designer

제품/서비스의 비즈니스 모델을 설계하는 워크플로우. 시장 데이터 기반으로 수익 구조, 가격 전략, 유닛 이코노믹스를 도출하고, 전략적 리스크를 검증한 뒤, AI가 소비할 수 있는 구조화된 BM 문서를 산출한다.

두 가지 모드로 동작한다:
- **독립 실행**: 사용자가 직접 호출. Phase 1(인터뷰)부터 시작.
- **파이프라인 모드**: 상위 워크플로우(예: Idea Forge)가 외부 컨텍스트를 주입. Phase 1을 건너뛰고 Phase 2부터 시작하되, 기존 리서치 데이터가 있으면 Phase 2도 축약.

## 워크플로우 전체 흐름

```
[진입 분기]
  ├─ 외부 컨텍스트 없음 → Phase 1부터 시작 (독립 실행)
  └─ 외부 컨텍스트 있음 → Phase 1 스킵
        ├─ 리서치 데이터 포함 → Phase 2 축약 (갭 분석만)
        └─ 리서치 데이터 없음 → Phase 2 전체 실행

Phase 1: 컨텍스트 수집 (사용자 인터뷰) — 외부 컨텍스트 시 스킵
    ↓
Phase 2: 시장 조사 (researcher 에이전트) — 기존 데이터 시 축약
    ↓
Phase 3: 수익 모델 설계
    ↓
Phase 4: 유닛 이코노믹스 산출
    ↓
Phase 5: 전략 검증 (cso 에이전트)
    ↓
Phase 6: BM Score 산출 (6개 지표 정량 평가)
    ↓
Phase 7: BM 문서 생성 (doc-writer-llm → doc-critic LLM 모드)
```

## 외부 컨텍스트 주입 인터페이스

상위 워크플로우가 BM Designer를 호출할 때, 아래 구조의 컨텍스트를 대화에 포함하여 전달한다. 이 컨텍스트가 존재하면 파이프라인 모드로 진입한다.

### 컨텍스트 구조

```
## BM Designer 외부 컨텍스트

### 제품 정보
- 제품/서비스 설명: {설명}
- 핵심 가치 제안: {가치}
- 타겟 고객: {고객 세그먼트}
- 차별화 포인트: {경쟁 우위}

### 검증 이력 (있으면)
- CEO 제안 요약: {제안 내용}
- CSO 검증 결과: {Accept/Rebuttal + 사유}

### 리서치 데이터 (있으면)
- 시장 규모: {TAM/SAM/SOM}
- 경쟁사 분석: {경쟁사별 가격/모델}
- 업종 벤치마크: {CAC, LTV, Churn, ARPU}
- 가격 민감도 데이터: {리뷰/포럼 기반 가격 인식, 가격 변경 이력}
```

### 진입 판단 규칙

1. **"제품 정보" 섹션이 존재하고 제품 설명 + 타겟 고객이 모두 채워져 있으면** → Phase 1 스킵. 필수 항목(현재 수익 상태, 경쟁 환경, 비용 구조, 제약 조건) 중 빠진 것이 있으면 해당 항목만 한 번의 질문으로 일괄 수집한 뒤 Phase 2로 진행.
2. **"리서치 데이터" 섹션이 존재하고 시장 규모 + 경쟁사 분석이 모두 채워져 있으면** → Phase 2를 축약 모드로 실행 (아래 Phase 2 참조).
3. **외부 컨텍스트 자체가 없으면** → 기존 워크플로우대로 Phase 1부터 시작.

## Phase 1: 컨텍스트 수집

> **파이프라인 모드**: 외부 컨텍스트의 "제품 정보" 섹션에 제품 설명 + 타겟 고객이 있으면, 이 Phase 전체를 건너뛴다. 필수 항목(현재 수익 상태, 경쟁 환경, 비용 구조, 제약 조건) 중 빠진 것이 있으면 해당 항목만 한 번의 질문으로 일괄 수집한 뒤 Phase 2로 진행한다. 전체 인터뷰를 다시 하지 않는다.

사용자에게 아래 항목을 인터뷰하라. 이미 대화에서 파악된 정보는 건너뛰되, 빠진 항목은 반드시 물어라. BM 설계의 품질은 여기서 수집하는 맥락의 정확도에 달려 있다.

### 필수 항목

| 항목 | 질문 예시 | 왜 필요한가 |
|------|----------|------------|
| 제품/서비스 설명 | "어떤 제품인가요? 핵심 가치는?" | 수익 모델의 출발점 |
| 타겟 고객 | "누가 쓰나요? B2B/B2C?" | 가격 민감도, 결제 방식 결정 |
| 현재 수익 상태 | "지금 매출이 있나요? 있다면 어떤 방식?" | 기존 모델 위에 설계할지 새로 시작할지 |
| 경쟁 환경 | "주요 경쟁사는? 차별점은?" | 가격 벤치마킹의 앵커 |
| 비용 구조 | "주요 비용은? (서버, 인건비, API 호출료, 인프라)" | 마진 계산, 손익분기점 산출 |
| 제약 조건 | "예산, 출시 일정, 규제 제약이 있나요?" | 실현 가능한 모델로 좁히기 |

### 선택 항목 (있으면 더 좋음)

- 현재 사용자 수 / 트래픽
- 투자 상태 (부트스트랩 / 시드 / 시리즈)
- 지역 (글로벌 / 특정 국가)
- 이전에 시도했다 실패한 모델

수집이 끝나면 "제가 이해한 내용을 정리하겠습니다" 라고 말하고, 수집 내용을 요약해서 사용자에게 확인을 받아라. 확인 없이 다음 단계로 넘어가지 마라.

## Phase 2: 시장 조사

**researcher 에이전트**를 호출하여 아래 데이터를 수집한다.

> **축약 모드** (파이프라인 모드에서 리서치 데이터가 이미 제공된 경우): 외부 컨텍스트의 "리서치 데이터" 섹션에 시장 규모 + 경쟁사 분석이 포함되어 있으면, 전체 조사를 다시 하지 않는다. 대신 아래 **갭 분석**만 수행한다:
>
> 1. 기존 리서치 데이터에서 아래 4개 조사 항목 중 누락되거나 불충분한 항목을 식별
> 2. 누락 항목이 있으면 해당 항목만 researcher에게 보충 조사 요청
> 3. 누락 항목이 없으면 기존 데이터를 그대로 사용하고 Phase 3으로 진행
>
> BM 설계에 특히 중요한 **경쟁사 가격/과금 모델**과 **업종별 벤치마크**가 빠져 있으면 반드시 보충한다. 시장 규모나 가격 민감도는 기존 데이터에 근거가 있으면 추가 조사 없이 진행해도 된다.

### 조사 항목

1. **경쟁사 가격/과금 모델** — 직접 경쟁사 3~5개의 가격표, 프리미엄 구분, 무료 티어 유무
2. **시장 규모** — TAM(전체), SAM(접근 가능), SOM(획득 가능) 추정치
3. **업종별 벤치마크** — 해당 업종의 평균 CAC, LTV, Churn rate, ARPU
4. **가격 민감도 신호** — 리뷰/포럼에서 "비싸다/싸다" 언급 트렌드, 경쟁사 가격 변경 이력

### researcher 호출 프롬프트 템플릿

```
다음 제품/서비스의 비즈니스 모델 설계를 위한 시장 조사를 수행하라:

타겟: {타겟 고객}

2. 시장 규모 (TAM/SAM/SOM, 출처 명시)
3. 업종 평균 지표 (CAC, LTV, Churn, ARPU)

```

```

## Phase 3: 수익 모델 설계

시장 조사 결과와 제품 특성을 바탕으로 수익 모델을 설계한다. 아래 레퍼런스에서 제품에 맞는 모델을 선택하고 조합하라.

### 수익 모델 레퍼런스

| 모델 | 적합한 경우 | 핵심 지표 | 리스크 |
|------|-----------|----------|--------|
| **구독 (Subscription)** | 반복 사용, 콘텐츠/SaaS | MRR, Churn, ARPU | Churn 관리 실패 시 성장 정체 |
| **프리미엄 (Freemium)** | 네트워크 효과, 바이럴 가능 | 전환율, 무료→유료 퍼널 | 무료 유저 비용 > 유료 수익 위험 |
| **종량제 (Usage-based)** | API, 클라우드, AI 서비스 | 사용량 성장률, 단가 | 예측 불가능한 매출 |
| **거래 수수료 (Transaction fee)** | 마켓플레이스, 결제 | GMV, Take rate | 양면 시장 구축 난이도 |
| **광고 (Advertising)** | 대규모 트래픽, 미디어 | DAU, CPM, 광고 fill rate | 광고주 의존, UX 훼손 |
| **인앱 구매 (IAP)** | 게임, 유틸리티 앱 | ARPPU, 구매율 | 소수 과금 유저 의존 |
| **라이선스 (License)** | 엔터프라이즈, B2B | 계약 규모, 갱신율 | 긴 세일즈 사이클 |
| **하이브리드** | 위 모델 2개 이상 조합 | 각 모델별 지표 | 복잡도 증가 |

### 설계 시 고려사항

- **가치 지표(Value Metric)를 먼저 정의하라**: 고객이 "이것 때문에 돈을 낸다"고 느끼는 단위 (예: 사용자 수, API 호출, 저장 용량, 생성 건수). 가치 지표와 과금 단위가 일치할수록 고객 납득도가 높다.
- **무료 티어 경계**: 무료로 제공할 범위는 "제품의 가치를 충분히 경험하되, 더 원하게 만드는 지점"에 설정한다. 너무 넓으면 전환 동기가 없고, 너무 좁으면 가치 체험 자체가 불가능하다.
- **가격 앵커링**: 경쟁사 가격 대비 자사 포지션을 명확히 하라 (저가 침투 / 동급 / 프리미엄).
- **단순함 우선**: 가격표가 복잡할수록 전환율이 떨어진다. 3개 이하의 플랜을 권장한다.

### 산출물

이 단계에서 아래 항목을 도출한다. 아래 템플릿을 그대로 사용하여 사용자에게 제시하라.

```

사용자에게 위 표를 보여주고 피드백을 받아라. 확인 없이 Phase 4로 넘어가지 마라.

## Phase 4: 유닛 이코노믹스

Phase 3에서 설계한 모델을 숫자로 검증한다. "이 모델이 실제로 돈이 되는가?"를 증명하는 단계다.

### 핵심 지표

| 지표 | 산식 | 건강한 기준 |
|------|------|-----------|
| **CAC** (고객 획득 비용) | 마케팅 비용 ÷ 신규 고객 수 | 업종별 벤치마크 참조 |
| **LTV** (고객 생애 가치) | ARPU × 평균 이용 기간 | LTV > 3× CAC |
| **LTV/CAC 비율** | LTV ÷ CAC | 3:1 이상 건강, 5:1 이상이면 투자 여력 |
| **Payback Period** | CAC ÷ 월 ARPU | 12개월 이내 권장 |
| **Gross Margin** | (매출 - 변동비) ÷ 매출 | SaaS 70%+, 앱 60%+, 마켓플레이스 40%+ |
| **Contribution Margin** | (매출 - 변동비 - 직접 고정비) ÷ 매출 | 양수여야 스케일 의미 있음 |
| **Break-even Point** | 고정비 ÷ 건당 공헌이익 | 현금 소진 전 도달 가능한지 |
| **Monthly Burn Rate** | 월 고정비 + 월 변동비 | 런웨이 계산용 |
| **Runway** | 보유 현금 ÷ Burn Rate | 12개월+ 권장 |

### 시나리오 모델링

세 가지 시나리오를 제시하라:

| 시나리오 | 가정 |
|---------|------|
| **보수적** | 전환율 하위 25%, CAC 상위 25%, Churn 상위 25% |
| **기본** | 업종 평균치 적용 |
| **낙관적** | 전환율 상위 25%, CAC 하위 25%, Churn 하위 25% |

각 시나리오별로 아래 템플릿의 12개월 프로젝션 표를 산출하라:

```
#### {시나리오명} 시나리오

| 월 | 신규 사용자 | 누적 사용자 | 유료 전환 | MRR | 변동비 | 고정비 | 순이익 | 누적 순이익 |
|----|-----------|-----------|----------|-----|--------|--------|--------|-----------|
| 1  | {수}       | {수}       | {수}      | {원} | {원}    | {원}    | {원}    | {원}       |
| 2  | ...       | ...       | ...      | ... | ...    | ...    | ...    | ...       |
| ...| ...       | ...       | ...      | ... | ...    | ...    | ...    | ...       |
| 12 | {수}       | {수}       | {수}      | {원} | {원}    | {원}    | {원}    | {원}       |
```

### 민감도 분석

수익에 가장 큰 영향을 미치는 변수 3개를 식별하고, 아래 템플릿으로 결과를 제시하라:

```
| 변수 | 기본값 | -20% 시 월순이익 | +20% 시 월순이익 | 변동 폭 |
|------|-------|-----------------|-----------------|---------|
| {변수1} | {값} | {원} | {원} | {원} |
| {변수2} | {값} | {원} | {원} | {원} |
| {변수3} | {값} | {원} | {원} | {원} |
```

## Phase 5: 전략 검증

**cso 에이전트**를 호출하여 설계한 BM의 리스크를 검증한다.

### cso 호출 프롬프트 템플릿

```
다음 비즈니스 모델의 전략적 타당성을 검증하라:

[Phase 3-4의 산출물 전체를 포함]

검증 항목:
1. 수익 모델의 지속 가능성 (경쟁사 대응, 시장 변화 시나리오)
2. 가격 전략의 적정성 (고객 지불 의향 대비)
3. 유닛 이코노믹스의 현실성 (가정이 낙관적이지 않은지)
4. 스케일링 시 비용 구조 변화 (규모의 경제 가능 여부)
5. 최대 리스크 3가지와 대응 방안
```

cso의 피드백을 받으면:
- **치명적 리스크** (모델 자체를 흔드는 것) → Phase 3로 돌아가 모델 수정
- **관리 가능한 리스크** → 대응 방안을 BM 문서에 포함
- **경미한 우려** → 모니터링 항목으로 기록

사용자에게 cso 검증 결과를 요약 보고하고, 수정이 필요하면 동의를 구하라.

## Phase 6: BM Score 산출

Phase 2~5의 산출물을 6개 지표로 정량 평가하여 BM의 종합 점수를 산출한다. 점수는 BM 문서에 포함되며, 약점 영역을 한눈에 식별하는 데 쓰인다.

### 6개 평가 지표 (각 0~10점)

| 지표 | 약어 | 평가 기준 | 데이터 출처 |
|------|------|----------|-----------|
| 시장 매력도 | MKT | TAM 규모, 성장률, 진입 장벽 높이 | Phase 2 시장 규모 데이터 |
| 수익 모델 적합도 | REV | 가치 지표와 과금 단위의 일치도, 가격 경쟁력, 플랜 단순성 | Phase 3 수익 모델 설계 |
| 유닛 이코노믹스 건전성 | UE | LTV/CAC 비율, Payback Period, Gross Margin의 업종 벤치마크 대비 수준 | Phase 4 핵심 지표 |
| 스케일 가능성 | SCL | 한계비용 체감 여부, 변동비 비율, 네트워크 효과 유무 | Phase 4 시나리오 모델링 |
| 전략적 방어력 | DEF | 경쟁 우위 지속성, 전환 비용, 모방 난이도 | Phase 5 cso 검증 결과 |
| 실행 리스크 | RSK | 상위 3 리스크의 심각도 역수. 리스크가 낮을수록 높은 점수 | Phase 5 리스크 평가 |

### 채점 기준

| 점수 | 기준 |
|------|------|
| 9-10 | 업종 상위 10% 수준 또는 명확한 경쟁 우위 확보 |
| 7-8 | 업종 평균 이상, 건강한 수준 |
| 5-6 | 업종 평균 수준, 개선 여지 있음 |
| 3-4 | 업종 평균 이하, 주의 필요 |
| 1-2 | 심각한 약점, 모델 재검토 필요 |
| 0 | 평가 불가 (해당 지표의 데이터 수집 실패 또는 해당 없음) |

0점 지표는 가중 평균 계산 시 해당 지표의 가중치를 나머지 지표에 비례 배분한다. 0점 지표가 3개 이상이면 BM Score를 산출하지 않고 "데이터 부족 — 추가 조사 필요"로 표시한다.

각 점수에는 1줄 근거를 반드시 기재한다. 근거 없는 점수는 부여하지 않는다.

### 종합 점수 산출

```
BM Score = (MKT × 0.15) + (REV × 0.20) + (UE × 0.25) + (SCL × 0.15) + (DEF × 0.15) + (RSK × 0.10)
```

가중치 근거:
- UE(유닛 이코노믹스)가 최고 가중치 — "돈이 되는가"가 BM의 핵심
- REV(수익 모델)가 두 번째 — 과금 구조가 시장과 맞아야 함
- MKT, SCL, DEF는 동일 가중치 — 시장, 확장성, 방어력은 균등하게 중요
- RSK가 최저 가중치 — 리스크는 대응 가능하므로 보조 지표

### 등급 판정

| BM Score | 등급 | 해석 |
|----------|------|------|
| 8.0 이상 | **A** | 강력한 BM. 즉시 실행 권장 |
| 6.5 ~ 7.9 | **B** | 양호한 BM. 약점 영역 보강 후 실행 |
| 5.0 ~ 6.4 | **C** | 보통. 2개 이상 지표 개선 필요 |
| 5.0 미만 | **D** | 재설계 필요. 모델 자체 재검토 권장 |

### 출력 템플릿

```
## BM Score

| 지표 | 점수 | 근거 |
|------|------|------|
| MKT (시장 매력도) | {0-10} | {1줄 근거} |
| REV (수익 모델 적합도) | {0-10} | {1줄 근거} |
| UE (유닛 이코노믹스) | {0-10} | {1줄 근거} |
| SCL (스케일 가능성) | {0-10} | {1줄 근거} |
| DEF (전략적 방어력) | {0-10} | {1줄 근거} |
| RSK (실행 리스크) | {0-10} | {1줄 근거} |

**BM Score: {가중 평균} / 10 | 등급: {A/B/C/D}**

약점 영역: {5점 이하 지표 나열, 없으면 "없음"}
우선 개선 항목: {가장 낮은 점수 지표의 개선 방향 1줄}
```

5점 이하 지표가 2개 이상이면 사용자에게 "해당 영역 보강 후 Phase 7(문서 생성) 진행을 권장합니다. 바로 진행하시겠습니까?"로 확인을 요청한다.

## Phase 7: BM 문서 생성

최종 BM 문서는 **AI가 소비하는 문서**다. doc-writer-llm가 작성하고, doc-critic(LLM 모드)이 검증한다.

### 메인 오케스트레이터에게 지시

이 단계에서 아래 순서를 따르라:

1. Phase 1~6의 모든 산출물(BM Score 포함)을 종합하여 doc-writer-llm에게 전달
2. doc-writer-llm가 아래 구조의 BM 문서를 작성
3. doc-critic (LLM 모드)이 검증
4. PASS 시 사용자에게 전달, REJECT 시 피드백 반영 후 재작성

### BM 문서 구조

doc-writer-llm에게 아래 구조를 지시하라:

```markdown
---
type: business-model
product: {제품명}
version: {버전}
date: {작성일}
status: validated
---

# Business Model: {제품명}

## 1. Product Overview
- 제품 설명, 핵심 가치, 타겟 고객

## 2. Market Context
- TAM/SAM/SOM, 경쟁 환경, 벤치마크 수치
- (출처 포함)

## 3. Revenue Model
- 선택 모델 및 근거
- 기각 대안 및 사유
- 가치 지표, 과금 단위

## 4. Pricing Strategy
- 가격 티어 (표)
- 무료/유료 경계 논리
- 경쟁사 대비 포지션

## 5. Unit Economics
- 핵심 지표 (CAC, LTV, LTV/CAC, Gross Margin, Payback Period)
- 3 시나리오 12개월 프로젝션
- 민감도 분석 결과

## 6. BM Score
- 6개 지표 점수표 (MKT, REV, UE, SCL, DEF, RSK)
- 종합 점수 + 등급 (A/B/C/D)
- 약점 영역 + 우선 개선 항목

## 7. Risk Assessment
- 상위 3 리스크 + 대응 방안
- 모니터링 지표

## 8. Action Items
- 즉시 실행 항목
- 검증 필요 가설
- 마일스톤별 KPI
```

### 저장 위치

`{project}/docs/bm/` 디렉토리에 저장하라. 파일명: `BM_PLAN.md` (기존 파일이 있으면 업데이트).

## 워크플로우 중단/재개

사용자가 중간에 중단하거나 특정 Phase만 요청할 수 있다:

- "시장 조사만 해줘" → Phase 2만 실행
- "가격 전략만 다시 짜줘" → Phase 3만 재실행 (기존 데이터 활용)
- "유닛 이코노믹스 검토해줘" → Phase 4만 실행 (기존 모델 기반)

어느 Phase든 독립적으로 실행 가능하되, 이전 Phase의 산출물이 없으면 필요한 정보를 사용자에게 물어라.

## 사용자 피드백 처리 규칙

Phase 1, Phase 3, Phase 5에서 사용자 확인을 요청한 뒤, 사용자가 수정을 요구하면 아래 절차를 따르라:

1. **특정 항목 수정 요구** ("이 부분 바꿔줘", "경쟁사 하나 더 추가해") → 해당 항목만 수정하여 다시 제시한다. Phase 전체를 재실행하지 않는다.
2. **전면 재작업 요구** ("전부 다시 해줘", "방향이 잘못됐어") → 해당 Phase를 처음부터 재실행한다.
3. **3회 연속 수정 요청** → "3회 수정했으나 만족스럽지 않은 것 같습니다. 어떤 방향이 맞는지 구체적으로 알려주시겠습니까?"로 명확한 방향을 요청한다.
4. **보류 응답** ("일단 넘어가자", "나중에 다시 보자") → 현재 산출물을 임시 확정으로 표시하고 다음 Phase로 진행한다. BM 문서에 "Phase N 산출물 미확정 — 재검토 필요" 경고를 포함한다.

---

## 범위 제한

### NEVER 규칙

1. NEVER 투자 추천, 매수/매도 판단, 특정 종목/자산에 대한 투자 조언을 제공하라. 사용자가 "이 사업에 투자할 만한가?" 류의 질문을 하면 "투자 판단은 이 스킬의 범위 밖입니다. BM 설계에 집중하겠습니다."로 응답하라.
2. NEVER 법률 자문(계약서 검토, 라이선스 법적 해석, 규제 준수 판단)을 수행하라. 규제 관련 제약은 Phase 1에서 사실로서 수집하되, 법적 판단이 필요한 항목은 "법률 전문가 확인 필요"로 표시하라.
3. NEVER 실제 결제 시스템 구현 또는 결제 API 연동 코드를 작성하라. 산출물은 BM 설계 문서(`BM_PLAN.md`)만이다.
4. NEVER Phase 간 전환 시 사용자 확인 없이 다음 Phase로 진행하라. Phase 1 요약 확인, Phase 3 산출물 피드백, Phase 5 수정 동의가 각각 필요하다.
5. NEVER 출처 없는 시장 데이터를 BM 문서에 포함하라. 모든 수치에는 출처(URL 또는 리포트명)가 있어야 한다.

### ALWAYS 규칙

1. ALWAYS Phase 2의 시장 데이터에 출처(URL 또는 리포트명)를 포함하라.
2. ALWAYS Phase 3 산출물을 사용자에게 표로 보여주고 피드백을 받은 뒤 Phase 4로 진행하라.
3. ALWAYS Phase 4에서 3개 시나리오(보수적, 기본, 낙관적)를 모두 산출하라. 하나의 시나리오만 제시하지 마라.

### 에러 처리

- **researcher 에이전트가 데이터 수집에 실패하면**: 실패한 조사 항목을 명시하고, 해당 항목 없이 진행 가능한지 사용자에게 판단을 요청한다. Phase 3의 수익 모델 설계에는 최소한 경쟁사 가격/과금 모델 데이터가 필요하다.
- **cso 에이전트 검증이 불가능하면** (에이전트 미응답, 타임아웃): Phase 5를 건너뛰되, BM 문서에 "전략 검증 미완료 — 별도 검증 필요" 경고를 포함한다.
- **외부 컨텍스트가 불완전하면** (제품 설명은 있으나 타겟 고객이 누락된 경우): 파이프라인 모드로 진입하지 않고, 누락 항목을 사용자에게 질문한 뒤 Phase 1 부분 실행으로 전환한다.
