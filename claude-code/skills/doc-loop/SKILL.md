# Doc Loop

컨텍스트 수집부터 품질 검증까지, 문서 작성의 전체 라이프사이클을 오케스트레이션하는 워크플로우.

## 왜 이 스킬이 필요한가

좋은 문서는 두 가지가 필요하다: (1) 작성자 머릿속 컨텍스트를 빠짐없이 끌어내는 것, (2) 결과물의 품질을 정량적으로 보장하는 것. 이 스킬은 사용자와의 협업적 컨텍스트 수집(Phase 1)과 자동 품질 루프(Phase 3)를 하나의 파이프라인으로 결합한다.

## 모드 분기

문서의 대상 독자에 따라 두 가지 경로로 분기한다:

| 대상 독자 | Writer 에이전트 | critic 모드 | 예시 |
|-----------|----------------|----------------|------|
| 사람 | `writer` (Human Docs 모드) | doc-human | README, 설계문서, API 문서, 온보딩 가이드, 변경로그, PRD, RFC, proposal |
| LLM | `writer` (LLM Docs 모드) | doc-llm | CLAUDE.md, 에이전트 정의(.md), 스킬 SKILL.md, 시스템 프롬프트 |

대상 독자가 불명확하면 사용자에게 "이 문서의 주요 독자는 사람인가요, LLM인가요?" 라고 확인한다.

## 이 스킬을 사용하지 않는 경우

- 실행 계획 요청 (예: "계획 세워줘", "리팩터링 플랜") → `plan-loop` 스킬 사용
- 단일 코드 작업 (예: "이 함수 고쳐줘", "테스트 추가해줘") → 직접 수행
- 단순 질문 (예: "이 코드 뭐 하는 거야?") → 직접 답변

## 워크플로우

### Phase 1: Context & Mode 결정

#### 1-1. 모드 결정

문서 유형을 보고 대상 독자(사람/LLM)를 판별한다. 불명확하면 사용자에게 확인한다.

#### 1-2. 작성 방식 선택

사용자에게 두 가지 방식을 제안한다:

> **A) 협업 모드**: 제가 질문하면서 컨텍스트를 함께 정리한 뒤 문서를 작성합니다. 머릿속에 정리되지 않은 내용이 많을 때 적합합니다.
> **B) 자동 모드**: 주신 요구사항으로 바로 초안을 작성하고 품질 루프를 돌립니다. 요구사항이 명확할 때 적합합니다.

- 사용자가 **A를 선택**하면 → Phase 1-3(Context Gathering)으로 진행
- 사용자가 **B를 선택**하면 → Phase 2(Drafting)로 바로 진행
- 사용자가 선택하지 않고 바로 요구사항을 던지면 → B로 간주하고 Phase 2로 진행

#### 1-3. Context Gathering (협업 모드 전용)

사용자의 머릿속 컨텍스트를 체계적으로 추출하는 단계.

**초기 질문** — 아래 5가지를 먼저 묻는다:
1. 문서 유형은? (기술 스펙, 의사결정 문서, 제안서 등)
2. 주요 독자는 누구인가?
3. 독자가 이 문서를 읽고 어떤 행동/판단을 하길 원하는가?
4. 따라야 할 템플릿이나 형식이 있는가?
5. 기타 제약사항이나 배경 컨텍스트는?

사용자가 짧게 답해도 된다고 안내한다.

**Info Dumping** — 초기 질문 후, 관련 컨텍스트를 모두 쏟아내도록 유도한다:
- 프로젝트/문제 배경
- 관련 팀 논의나 문서
- 대안이 기각된 이유
- 조직 컨텍스트 (팀 역학, 과거 사건)
- 타임라인, 기술 아키텍처, 이해관계자 우려

정리 안 해도 된다고 안내한다. 스트림으로 던져도 되고, 파일/링크를 공유해도 된다.

**Clarifying Questions** — 사용자가 초기 덤프를 마치면, 빈틈을 채우는 후속 질문 5-10개를 던진다.

**종료 조건**: 엣지 케이스와 트레이드오프를 물을 수 있을 만큼 컨텍스트가 충분히 모이면 Phase 2로 전환한다. 전환 전 사용자에게 "더 추가할 내용이 있나요, 아니면 초안 작성으로 넘어갈까요?" 라고 확인한다.

### Phase 2: Drafting

모드에 따라 적절한 Writer 에이전트를 호출하여 문서 초안을 받는다.

- **사람용**: `writer` 에이전트(subagent_type: `writer`, Human Docs 모드) 호출
- **LLM용**: `writer` 에이전트(subagent_type: `writer`, LLM Docs 모드) 호출
- Phase 1에서 수집한 컨텍스트, 문서 목적, 대상 독자 정보를 Writer에게 전달

### Phase 3: Quality Loop

#### 3-1. 초안 검증 게이트

critic에게 보내기 전에 최소 품질을 확인한다.

**즉시 REJECT 조건** (하나라도 해당하면):
- 문서가 비어 있음
- 본문이 3문장 이하

REJECT 시: Writer에게 아래 메시지와 함께 재작성을 요청하고 Phase 2로 복귀한다.
> "출력이 불충분합니다. 최소 요구사항을 충족하는 초안을 작성하세요."

검증 게이트를 통과하면 3-2단계로 진행한다.

#### 3-2. Parity Check (코드 정합성 검증)

doc-parity-checker 에이전트(subagent_type: `doc-parity-checker`)를 호출하여 문서 내 코드 참조가 실제 코드베이스와 일치하는지 검증한다.

검증 대상: 파일 경로, 함수/클래스 시그니처, CLI 명령어, 환경변수, 의존성, 디렉토리 구조

**결과 처리:**
- **MISMATCH**: 불일치 목록을 Writer에게 전달하여 수정 요청 → Phase 2로 복귀. critic을 호출하지 않는다.
- **MATCH**: 3-3단계(critic)로 진행한다.

#### 3-3. critic 채점

critic 에이전트(subagent_type: `critic`)를 호출한다. Phase 1에서 선택한 모드를 critic에게 전달한다.

**doc-human 모드 채점 기준:**
| 기준 | 가중치 |
|------|--------|
| Readability | 30% |
| Structure | 25% |
| Examples | 20% |
| Completeness | 15% |
| Accuracy | 10% |

**doc-llm 모드 채점 기준:**
| 기준 | 가중치 |
|------|--------|
| Precision | 30% |
| Executability | 25% |
| Boundary Clarity | 20% |
| Consistency | 15% |
| Edge Cases | 10% |

- **PASS 기준**: 총점 > 8.00 AND 최고 가중치 기준(Readability 또는 Precision) >= 8
- 라운드당 **최대 3개**의 피드백 — 가장 낮은 점수 기준부터 순서대로 지적 (REJECT 시에만)

#### 3-4. 결과 처리

- **REJECT**: critic의 피드백을 Writer에게 전달하여 수정 요청 → 3-1단계로 복귀
- **PASS**: 최종 문서를 사용자에게 전달하고 루프 종료

## 라운드 보고

매 라운드마다 사용자에게 아래 형식으로 진행 상황을 보고한다:

```
[라운드 N] 점수: X.XX | 결과: PASS/REJECT | 피드백: (1줄 요약)
```

## NEVER 규칙

1. NEVER: 문서가 아닌 산출물(실행 계획, 코드, 답변)을 이 루프로 처리하지 마라. 실행 계획은 `plan-loop` 스킬을, 코드와 답변은 직접 수행을 사용한다.
2. NEVER: 초안 검증 게이트(3-1단계)를 건너뛰고 critic을 호출하지 마라. 모든 Writer 출력은 반드시 게이트를 통과해야 한다.
3. NEVER: Writer 에이전트가 critic이나 doc-parity-checker를 직접 호출하도록 허용하지 마라. 모든 에이전트 호출은 메인 모델만 수행한다.
4. NEVER: 사람용 문서에 LLM 모드를, LLM용 문서에 HUMAN 모드를 적용하지 마라. 모드 불일치는 채점 기준이 맞지 않아 무의미한 피드백을 생성한다.
5. NEVER: Parity Check(3-2단계)에서 MISMATCH가 발견된 상태로 critic(3-3단계)을 호출하지 마라. 코드 정합성이 확보된 후에만 품질 채점을 진행한다.

## 제한 규칙

- **최대 반복 횟수: 5회.** 검증 게이트 REJECT도 횟수에 포함한다. 5회 연속 REJECT 시 현재까지 가장 높은 점수를 받은 문서와 미해결 이슈를 사용자에게 보고하고 판단을 요청한다.

## 엣지 케이스

| 상황 | 처리 |
|------|------|
| Writer가 에러를 반환하거나 응답 없음 | 1회 재시도한다. 재시도도 실패하면 사용자에게 "Writer 호출에 실패했습니다. 요청을 다시 시도하시겠습니까?" 라고 보고하고 루프를 종료한다. |
| critic이 에러를 반환하거나 응답 없음 | 1회 재시도한다. 재시도도 실패하면 현재 문서를 사용자에게 전달하며 "critic 채점 불가. 수동 검토가 필요합니다." 라고 안내한다. |
| 대상 독자가 불명확 (사람/LLM 구분 불가) | Writer를 호출하지 않는다. 사용자에게 "이 문서의 주요 독자는 사람인가요, LLM인가요?" 라고 되묻는다. |
| Writer가 이전 라운드와 동일한 문서를 반환 (critic 피드백 미반영) | 해당 라운드를 REJECT로 처리한다. critic 피드백 원문을 재인용하여 "이전 피드백이 반영되지 않았습니다. 아래 피드백을 반영하여 수정하세요:" 메시지와 함께 Writer에게 재전달한다. |
| doc-parity-checker가 에러를 반환하거나 응답 없음 | 1회 재시도한다. 재시도도 실패하면 parity check를 건너뛰고 critic으로 진행한다. 사용자에게 "parity check 실패. 코드 정합성 미검증 상태입니다." 라고 안내한다. |
| critic이 PASS를 반환했으나 총점이 8.00 이하 (채점 오류 의심) | PASS를 무시하고 REJECT로 처리한다. critic에게 재채점을 요청한다. |
| 5라운드 소진 시 최선 문서 선택 | 5라운드 중 가장 높은 총점을 받은 문서를 사용자에게 전달한다. 모든 라운드의 점수를 함께 보고한다. |
| 사용자가 루프 도중 중단을 요청 | 즉시 루프를 중단한다. 현재까지의 최선 문서를 사용자에게 전달한다. |
| 협업 모드에서 사용자가 도중에 "그냥 알아서 써줘" | 현재까지 수집된 컨텍스트로 Phase 2(자동 모드)로 전환한다. |
