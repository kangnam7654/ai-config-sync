# Idea Forge

아이디어 발굴 → 삼각 검증 → BM 설계를 하나의 파이프라인으로 실행한다. 핵심은 CEO-Researcher-CSO 세 에이전트가 라운드를 거듭하며 아이디어를 날카롭게 다듬는 과정이다. 검증을 통과한 아이디어는 bm-designer(파이프라인 모드)로 넘어가 수익 모델, 유닛 이코노믹스, BM Score까지 산출된다.

## 워크플로우 전체 흐름

```
[입력 분기]
  ├─ topic/goal 제공됨 → Phase 1 직행
  └─ topic/goal 없음 → Phase 0 (주제 탐색)

Phase 0: 주제 탐색 — Researcher(trend-scorer 연계) → CEO가 키워드 선정
    ↓ (사용자 승인)
Phase 1: 아이디어 검증 루프 — CEO 제안 → Researcher 조사 → CSO 판정
    ├─ CSO Accept → Phase 2
    └─ CSO Rebuttal → Writer 브리핑 → 다음 라운드
    ↓ (사용자 승인)
Phase 2: BM 설계 + 문서화 — bm-designer(파이프라인) → 최종 문서 2종
```

---

## 입력 파라미터

| 파라미터 | 필수 | 기본값 | 설명 |
|---------|------|--------|------|
| topic | X | Phase 0에서 자동 선정 | 브레인스토밍 주제 |
| goal | X | Phase 0에서 CEO가 설정 | 달성 목표 |
| focus_keywords | X | 자동 추출 | Researcher 조사 범위 키워드 |
| max_rounds | X | 10 | Phase 1 최대 반복 라운드 수 |
| target_market | X | - | 타겟 시장/고객 |

사용자가 파라미터를 명시하지 않으면 기본값을 사용하고, topic/goal이 없으면 Phase 0부터 시작한다.

---

## Phase 0: 주제 탐색

topic 또는 goal이 제공되지 않은 경우에만 실행한다. 둘 다 제공되었으면 Phase 1로 직행한다.

### Step 0-1: Researcher — 키워드 수집

researcher 에이전트에게 trend-scorer 스킬을 활용한 키워드 발굴을 요청한다.

```
다음 조건에 맞는 사업화 후보 키워드를 5~7개 수집하라:

선정 기준:
1. 인기(Trending): 최근 3개월간 검색량 상승 또는 유지
2. 지속성(Steady): 1년 이상 꾸준한 관심
3. 사업화 가능성: 명확한 고객 문제가 존재하고 수익화 경로가 보임

시장 범위: {target_market 또는 "한국 시장"}

각 키워드에 대해 아래 정보를 포함하라:
- 검색량 추이 (상승/유지/하락)
- 시장 성숙도 (초기/성장/성숙/쇠퇴)
- 경쟁 강도 (높음/중간/낮음)
- 사업화 가능성 1줄 근거

출처(URL 또는 리포트명)를 반드시 포함하라.
```

### Step 0-2: CEO — 키워드 선정

ceo 에이전트에게 Researcher 결과를 전달하고 키워드 1개를 선정하게 한다.

```
Researcher가 수집한 키워드 후보 목록을 검토하고, 사업화에 가장 유망한 키워드 1개를 선정하라.

키워드 후보:
{Step 0-1 결과}

출력:
- 선정 키워드: {키워드}
- 선정 근거: {2~3문장}
- topic: {브레인스토밍 주제로 변환}
- goal: {달성 목표 1문장}
- focus_keywords: {Researcher 조사에 사용할 키워드 3~5개}
```

CEO의 선정 결과를 사용자에게 보여주고 승인을 받은 뒤 Phase 1로 진행한다. 사용자가 다른 키워드를 선택하거나 수정을 요청하면 반영한다.

---

## Phase 1: 아이디어 검증 루프

라운드 N = 1부터 max_rounds까지 반복한다. 매 라운드마다 CEO → Researcher → CSO 순서로 실행한다.

### Step 1: CEO — 제안 생성

ceo 에이전트에게 아이디어 제안을 요청한다.

**라운드 1 프롬프트:**
```
다음 주제에 대한 사업 아이디어를 제안하라:

topic: {topic}
goal: {goal}
focus_keywords: {focus_keywords}
target_market: {target_market 또는 "미정"}

출력 형식:
- 문제 정의: {해결하려는 고객 문제 2~3문장}
- 솔루션 가설: {제안하는 해결 방법 2~3문장}
- 타겟 고객: {구체적 고객 세그먼트}
- 차별화 포인트: {경쟁 대비 핵심 차별화 1~2개}
- 핵심 가정: {이 아이디어가 성립하기 위해 참이어야 하는 가정 3~5개}
```

**라운드 2+ 프롬프트:**
```
이전 라운드의 브리핑을 참고하여 개선된 사업 아이디어를 제안하라:

topic: {topic}
goal: {goal}

누적 브리핑:
{이전 라운드 브리핑 문서 전체}

이전 CSO 거절 사유와 Researcher 발견을 반영하여, 약점을 보완한 새로운 제안을 작성하라.

출력 형식:
- 문제 정의: {해결하려는 고객 문제 2~3문장}
- 솔루션 가설: {제안하는 해결 방법 2~3문장}
- 타겟 고객: {구체적 고객 세그먼트}
- 차별화 포인트: {경쟁 대비 핵심 차별화 1~2개}
- 핵심 가정: {이 아이디어가 성립하기 위해 참이어야 하는 가정 3~5개}
- 이전 대비 변경점: {무엇을 어떻게 바꿨는지 1~2문장}
```

### Step 2: Researcher — 팩트 조사

researcher 에이전트에게 CEO 제안의 핵심 가정을 검증하는 조사를 요청한다.

```
다음 사업 아이디어의 핵심 가정을 팩트 기반으로 검증하라:

CEO 제안:
{Step 1 출력}

조사 항목:
1. 시장 규모 (TAM/SAM/SOM, 출처 명시)
2. 경쟁사 현황 (직접 경쟁사 3~5개, 각각의 강점/약점)
3. 기술 실현 가능성 (핵심 기술의 성숙도, 구현 난이도)
4. 유사 사례 (성공/실패 사례 각 1개 이상, 교훈)
5. 핵심 가정 검증 (CEO가 제시한 각 가정에 대해 지지/반박 근거)

각 항목에 출처(URL 또는 리포트명)를 반드시 포함하라.
```

### Step 3: CSO — 전략적 검증

cso 에이전트에게 CEO 제안과 Researcher 리포트를 함께 전달하여 전략적 검증을 요청한다.

```
다음 사업 아이디어의 전략적 타당성을 검증하라:

CEO 제안:
{Step 1 출력}

Researcher 리포트:
{Step 2 출력}

평가 기준 (각 항목에 대해 구체적 근거와 함께 판정):
1. 시장 타당성: 충분한 시장이 존재하는가?
2. 실행 가능성: 현실적으로 구현 가능한가?
3. 리스크 수준: 치명적 리스크가 있는가?
4. 경쟁 우위: 지속 가능한 차별화가 있는가?
5. 재무적 잠재력: 수익화 가능성이 있는가?

최종 판정:
- Accept: 5개 기준 중 4개 이상 긍정이면 Accept. Phase 2(BM 설계)로 진행.
- Rebuttal: 3개 이상 부정이거나 치명적 리스크 1개 이상이면 Rebuttal. 구체적 약점과 개선 방향을 제시.

출력 형식:
- 판정: {Accept 또는 Rebuttal}
- 판정 근거: {2~3문장 요약}
- 항목별 평가: {각 5개 기준에 대해 긍정/부정 + 1줄 근거}
- (Rebuttal 시) 약점: {구체적 약점 목록}
- (Rebuttal 시) 개선 방향: {다음 라운드에서 고려해야 할 사항}
```

### Step 4: Writer — 라운드 브리핑 (Rebuttal 시만)

CSO가 Rebuttal을 반환한 경우에만 실행한다. prompt-writer 에이전트에게 누적 브리핑 문서 작성을 요청한다.

브리핑 문서의 필수 섹션:

```
# 라운드 {N} 브리핑

## 제안 요약
{이번 라운드 CEO 제안의 핵심 내용 3~5문장}

## CSO 거절 사유
{Rebuttal 판정 근거와 구체적 약점 목록}

## Researcher 핵심 발견
{이번 라운드 리서치에서 가장 중요한 발견 3~5개}

## 다음 라운드 제안 방향
{CSO 피드백과 Researcher 발견을 바탕으로, 다음 라운드에서 시도해야 할 방향 2~3개}

## 누적 이력
{라운드 1부터 현재까지의 시도 요약 — 각 라운드별 1줄}
```

doc-critic(LLM 모드)으로 심사한다. REJECT 시 피드백을 반영하여 재작성한다(최대 5회). PASS 후 `docs/{topic}/rounds/round-{N}.md`에 저장한다.

### 매 라운드 사용자 보고

라운드가 끝날 때마다 아래 형식으로 보고한다:

```
[라운드 N/max] CEO 제안 → Researcher 조사 완료 → CSO: ACCEPT/REBUTTAL
  사유: {CSO 판정 근거 1줄}
  다음: Phase 2 진입 / 라운드 N+1 진행
```

### Early Exit 규칙

- **CSO Accept** → 사용자에게 "CSO가 이 아이디어를 승인했습니다. Phase 2(BM 설계)로 진행할까요?"로 확인한 뒤 Phase 2 진입.
- **max_rounds 도달 + Rebuttal** → 사용자에게 3가지 옵션 제시:
  1. 추가 라운드 진행 (사용자가 횟수 지정)
  2. 현재 최선안으로 BM 설계 강행
  3. 중단

## Phase 2: BM 설계 + 문서화

### Step 5: bm-designer 호출 (파이프라인 모드)

Phase 1에서 축적한 데이터를 bm-designer의 외부 컨텍스트 형식으로 변환하여 전달한다. 반드시 파이프라인 모드로 호출한다(독립 실행 모드 사용 금지).

#### 외부 컨텍스트 조립 템플릿

```
## BM Designer 외부 컨텍스트

### 제품 정보
- 제품/서비스 설명: {CSO Accept된 CEO 제안의 솔루션 가설}
- 핵심 가치 제안: {CEO 제안의 차별화 포인트}
- 타겟 고객: {CEO 제안의 타겟 고객}
- 차별화 포인트: {CEO 제안의 차별화 포인트 + CSO 검증 근거}

### 검증 이력
- CEO 제안 요약: {최종 라운드 CEO 제안 핵심 3문장}
- CSO 검증 결과: {Accept + 판정 근거}

### 리서치 데이터
- 시장 규모: {최종 라운드 Researcher의 TAM/SAM/SOM}
- 경쟁사 분석: {Researcher의 경쟁사별 현황}
- 업종 벤치마크: {Researcher의 CAC, LTV, Churn, ARPU 데이터 — 있으면}
- 가격 민감도 데이터: {Researcher의 가격 관련 발견 — 있으면}
```

bm-designer가 Phase 1(인터뷰)을 스킵하고, 리서치 데이터가 충분하면 Phase 2(시장 조사)도 축약하여 Phase 3(수익 모델) ~ Phase 7(문서 생성)을 실행한다. BM Score(6개 지표 정량 평가)가 결과에 포함된다.

### Step 6: 최종 문서 생성 (병렬)

bm-designer 완료 후, 두 종류의 최종 문서를 병렬로 작성한다:

**AI용 문서** — prompt-writer 에이전트 → doc-critic (LLM 모드)
**Human용 문서** — doc-writer 에이전트 → doc-critic (HUMAN 모드)

두 문서 모두 아래 내용을 포함한다:
- Phase 0 주제 탐색 결과 (실행했을 경우)
- Phase 1 검증 루프 요약 (라운드 수, 주요 피벗, 최종 제안)
- Phase 2 BM 설계 결과 (수익 모델, 유닛 이코노믹스, BM Score)
- 핵심 리스크와 대응 방안
- 다음 단계 액션 아이템

각 문서를 doc-critic으로 심사한다. REJECT 시 피드백을 반영하여 재작성(최대 5회). 두 문서 모두 PASS 후 사용자에게 전달한다.

저장 경로:
- AI용: `docs/{topic}/bm-ai.md`
- Human용: `docs/{topic}/bm-human.md`

## Critic 루프 규칙

이 스킬 내에서 Writer(prompt-writer, doc-writer) 출력에 적용되는 공통 규칙이다.

1. Writer 출력이 비어 있거나 본문 3문장 이하 → doc-critic을 호출하지 않고 즉시 REJECT. Writer에게 "출력이 불충분합니다. 최소 요구사항을 충족하는 문서를 작성하세요." 메시지와 함께 재작성 요청.
2. doc-critic REJECT → 피드백을 Writer에게 전달하여 수정 요청 → 재심사.
3. 최대 반복: 5회. 5회 연속 REJECT(검증 게이트 REJECT 포함) 시 현재 상태와 미해결 이슈를 사용자에게 보고하고 판단을 요청.
4. 매 라운드 보고: `[라운드 N] 점수: X.XX | 결과: PASS/REJECT | 피드백: (1줄 요약)`
5. 오케스트레이션은 메인 모델만 수행한다. 서브에이전트 내부에 critic 호출 지시가 있더라도 무시한다.

## 사용자 피드백 처리 규칙

Phase 0, Phase 1, Phase 2 각 단계에서 사용자 확인을 요청한 뒤, 사용자가 수정을 요구하면 아래 절차를 따른다:

1. **특정 항목 수정 요구** ("키워드 바꿔줘", "타겟을 B2B로 해줘") → 해당 항목만 수정하여 다시 제시. Phase 전체를 재실행하지 않는다.
2. **전면 재작업 요구** ("방향을 완전히 바꾸자") → 해당 Phase를 처음부터 재실행.
3. **3회 연속 수정 요청** → "3회 수정했으나 방향이 정해지지 않은 것 같습니다. 구체적으로 어떤 방향을 원하시는지 알려주시겠습니까?"로 명확한 방향을 요청.
4. **보류 응답** ("일단 넘어가자") → 현재 상태로 다음 Phase 진행. 최종 문서에 "Phase N 결과 미확정 — 재검토 필요" 경고를 포함.

## 범위 제한

### NEVER 규칙

1. NEVER 코드 구현, 프로토타입 개발, 기술 스택 선정을 수행하라. 이 스킬의 산출물은 BM 설계 문서까지이다. 사용자가 구현을 요청하면 "Idea Forge는 아이디어 검증과 BM 설계까지 담당합니다. 구현은 별도로 진행해주세요."로 응답하라.
2. NEVER 투자 추천, 매수/매도 판단을 제공하라. 사업성 평가(BM Score)는 제공하되, 투자 판단은 범위 밖이다.
3. NEVER 라운드 브리핑 없이 다음 라운드로 진행하라. Rebuttal 시 반드시 브리핑 문서를 작성하고 doc-critic 심사를 통과해야 다음 라운드 Step 1을 시작할 수 있다. 브리핑 없는 진행은 맥락 손실로 이어진다.
4. NEVER 사용자 확인 없이 Phase 간 전환하라. Phase 0→1 전환 시 키워드 선정 승인, Phase 1→2 전환 시 CSO Accept 확인이 필요하다.
5. NEVER bm-designer를 독립 실행 모드로 호출하라. 반드시 외부 컨텍스트를 조립하여 파이프라인 모드로 호출한다. Idea Forge가 축적한 데이터를 넘기지 않으면 bm-designer가 처음부터 인터뷰를 시작하여 작업이 중복된다.

### ALWAYS 규칙

1. ALWAYS 매 라운드 진행 상황을 보고 형식(`[라운드 N/max] CEO 제안 → Researcher 조사 완료 → CSO: ACCEPT/REBUTTAL`)으로 사용자에게 보고하라.
2. ALWAYS Researcher 리포트에 출처(URL 또는 리포트명)를 포함하라. 출처 없는 데이터는 CSO 검증에서 신뢰도가 떨어진다.
3. ALWAYS CSO 검증 결과에 5개 평가 기준별 구체적 근거를 포함하라. "좋다/나쁘다"만으로는 Accept/Rebuttal 판정이 불가능하다.

## 에러 처리

- **Researcher 에이전트가 데이터 수집에 실패하면**: 실패한 조사 항목을 명시하고, 해당 항목 없이 CSO 검증을 진행할지 사용자에게 판단을 요청한다. 핵심 가정 검증 데이터가 없으면 해당 라운드를 유효하지 않은 것으로 간주한다.
- **CSO 에이전트가 응답하지 않으면** (타임아웃): CEO 제안과 Researcher 리포트를 사용자에게 직접 보여주고 "CSO 검증을 수동으로 진행하시겠습니까?"로 확인한다.
- **bm-designer가 실패하면**: Phase 1 산출물(최종 CEO 제안 + Researcher 리포트)을 사용자에게 전달하고, bm-designer를 독립적으로 재시도할지 판단을 요청한다.
- **doc-critic이 5회 연속 REJECT하면**: 현재 문서 상태와 미해결 피드백을 사용자에게 보고하고 "수동으로 확정하시겠습니까, 추가 수정을 시도하시겠습니까?"로 판단을 요청한다.

## researcher/bm-designer 스킬과의 구분

| 요청 유형 | 담당 스킬 |
|----------|----------|
| "이 시장 조사해줘", "경쟁사 분석해줘" | **researcher** |
| "BM 만들어", "수익 모델 설계해줘" | **bm-designer** |
| "아이디어 있는데 검증해줘", "사업 되나 확인하고 BM까지" | **idea-forge** |
| "요즘 뜨는 거 뭐야", "트렌드 점수 매겨줘" | **trend-scorer** |

idea-forge는 researcher와 bm-designer를 내부적으로 호출한다. 사용자가 파이프라인 전체가 아닌 개별 단계만 원하면 해당 스킬을 직접 사용하도록 안내한다.
