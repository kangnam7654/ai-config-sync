# Idea Forge

auto-dev 파이프라인의 **Idea Phase (#1~#9)**를 오케스트레이션한다. 사용자의 한 문장 입력을 받아 검증된 아이디어 + Idea Brief 문서까지 산출한다. `--bm` 플래그 시 BM 설계(#7)를 추가 수행한다.

## 워크플로우 전체 흐름

```
#1 입력 분류 (메인 모델)
  ├─ "같이 고민" → #2 브레인스토밍 (CEO + 사용자)
  ├─ "막연한 요청" → #3 트렌드 스코어링 (researcher)
  └─ "아이디어 있음" → #4 CEO 방향 결정

#2 브레인스토밍 → #3 or #4
#3 트렌드 스코어링 → #4
#4 CEO 방향 결정 → #5 시장 조사 (researcher)
#5 시장 조사 → #6 전략 검증 (CSO)
#6 전략 검증
  ├─ No-Go → #4 복귀 (최대 10회)
  └─ Go → --bm? #7 BM 설계 : #8 Idea 게이트
#7 BM 설계 (--bm 시에만) → #8 Idea 게이트 (CEO)
#8 Idea 게이트
  ├─ BM 수정 → #7 (--bm 시에만)
  ├─ 방향 변경 → #4
  └─ PASS → #9 Idea Brief 문서화 (doc-loop)
#9 Idea Brief 문서화 → Idea Phase 완료
```

---

## #1 입력 분류

메인 모델이 아래 판별 규칙을 **우선순위 순서대로** 적용한다. 첫 번째로 매칭되는 유형으로 확정한다.

| 우선순위 | 유형 | 판별 조건 | 예시 | 시작 |
|---|---|---|---|---|
| 1 | 같이 고민 | 질문형("~할까", "~좋을까", "어떤 게") 또는 협의 요청("같이", "고민", "추천") | "뭐가 좋을까", "같이 고민해줘" | #2 |
| 2 | 막연한 요청 | 구체적 앱 이름/기능 없이 목적/조건만 제시 | "돈 되는 앱 만들어", "요즘 뜨는 거" | #3 |
| 3 | 아이디어 있음 | 구체적 앱 유형, 이름, 핵심 기능 1개 이상 명시 | "할일 앱 만들어", "레시피 SNS" | #4 |

**매칭 실패 시**: 사용자에게 "아이디어가 이미 정해져 있나요, 함께 고민하고 싶으신가요?" 확인 후 분류.

---

## #2 브레인스토밍 (선택)

"같이 고민" 유형일 때만 실행. **CEO 에이전트 + 사용자**가 협력하여 아이디어를 발굴한다.

1. CEO 에이전트에게 사용자의 관심사/조건을 전달하여 아이디어 3~5개를 제안받는다
2. 사용자에게 CEO의 제안을 보여주고 선택/수정/추가 의견을 받는다
3. 확정된 아이디어로 #3(트렌드 확인) 또는 #4(바로 정제)로 진행

**산출물**: brainstorming.yaml 형식

---

## #3 트렌드 스코어링 (선택)

"막연한 요청" 유형이거나, #2에서 트렌드 확인이 필요할 때 실행.

**researcher 에이전트**를 트렌드 스코어링 모드로 호출한다:

```
아래 파라미터로 트렌드 스코어링을 수행하라:
- 주제: {사용자 관심 분야 또는 "자동 발굴"}
- 시장: {한국/글로벌}
- 카테고리: {지정 카테고리 또는 범용}
- 가중치: α=0.55, β=0.45 (기본값)
```

researcher 결과(Trend Score Report)를 CEO에게 전달하여 #4에서 방향 결정에 활용.

**산출물**: trend-scoring.yaml 형식

---

## #4 CEO 방향 결정

**CEO 에이전트**에게 이전 단계 결과를 전달하여 최종 방향을 결정한다:

```
아래 정보를 바탕으로 앱/서비스 방향을 결정하라:

{#2 브레인스토밍 결과 (있으면)}
{#3 트렌드 스코어링 결과 (있으면)}
{사용자 원본 입력}

출력:
- 앱 이름
- 한 문장 요약
- 타겟 사용자
- 핵심 기능 3개
- 차별점
```

**산출물**: ceo-direction.yaml 형식

---

## #5 시장 조사

**researcher 에이전트**에게 #4의 방향을 전달하여 시장 데이터를 수집한다:

```
다음 앱/서비스의 시장 조사를 수행하라:

앱: {#4 앱 이름}
타겟: {#4 타겟 사용자}

조사 항목:
1. 시장 규모 (TAM/SAM/SOM, 출처 포함)
2. 경쟁사 3~5개 (강점, 약점, 가격 모델)
3. 기회 요약 (1~2문장)
4. 리스크 요약 (1~2문장)
```

**산출물**: market-research.yaml 형식

---

## #6 전략 검증

**CSO 에이전트**에게 #4 + #5 결과를 전달하여 전략적 타당성을 검증한다:

```
다음 앱 아이디어의 전략적 타당성을 검증하라:

{#4 CEO 방향 결정 전체}
{#5 시장 조사 전체}

검증 항목:
1. 시장 진입 타이밍
2. 경쟁 우위 지속 가능성
3. 타겟 시장 규모의 현실성
4. 최대 리스크 3가지와 대응 방안
```

CSO verdict 처리:
- **Go** → #7 BM 설계로 진행
- **No-Go** → #4 CEO 방향 결정으로 복귀. CSO 피드백을 CEO에게 전달하여 방향 수정. **최대 10회 반복**. 10회 소진 시 사용자에게 보고: "전략 검증 10회 실패. 아이디어 근본적 재검토 필요."

**산출물**: review-verdict.yaml 형식

---

## #7 BM 설계 (`--bm` 시에만 실행)

`--bm` 플래그가 없으면 이 단계를 SKIP하고 #8로 직행한다.

**bm-designer 에이전트**를 호출한다. #4~#6의 컨텍스트를 전달:

```
아래 컨텍스트로 BM을 설계하라:

## 제품 정보
{#4 CEO 방향 결정}

## 시장 데이터
{#5 시장 조사 결과}

## 전략 검증
{#6 CSO Go 판정}

수익 모델 선택, 가격 티어 설계, 유닛 이코노믹스(3시나리오), BM Score를 산출하라.
```

**산출물**: bm-design.yaml 형식

---

## #8 Idea 게이트 (CEO)

**CEO 에이전트**에게 이전 단계의 전체 결과를 전달하여 최종 의사결정.

### `--bm` 시 (BM 포함)

```
Idea Phase의 전체 산출물을 검토하고 최종 의사결정을 내려라:

{#4 방향 결정}
{#5 시장 조사}
{#6 전략 검증}
{#7 BM 설계 + BM Score}

의사결정 옵션:
- PASS: 아이디어와 BM이 충분. Design Phase로 진행.
- BM_REVISION: BM이 부족. 구체적 수정 지시와 함께 #7로 복귀.
- DIRECTION_CHANGE: 아이디어 자체 변경 필요. 새로운 방향과 함께 #4로 복귀.
```

### 기본 (BM 없이)

```
Idea Phase의 전체 산출물을 검토하고 최종 의사결정을 내려라:

{#4 방향 결정}
{#5 시장 조사}
{#6 전략 검증}

의사결정 옵션:
- PASS: 아이디어가 충분. Design Phase로 진행.
- DIRECTION_CHANGE: 아이디어 자체 변경 필요. 새로운 방향과 함께 #4로 복귀.
```

**최대 10회 반복**. 10회 소진 시 사용자에게 보고: "Idea Phase 10회 반복 실패."

**산출물**: gate-decision.yaml 형식

---

## #9 Idea Brief 문서화

**doc-loop 스킬**을 자동(B) 모드 + LLM 모드로 호출한다. #4~#8의 모든 산출물을 컨텍스트로 전달하여 `idea-brief.md`를 생성한다.

doc-loop 내부에서 doc-writer-llm → parity check → critic 루프가 PASS할 때까지 최대 5라운드 반복한다.

**산출물**: `{project}/docs/llm/idea-brief.md`

문서 내용:
- 프로젝트 개요 (앱명, 타겟, 핵심기능, 차별점)
- 시장 분석 (TAM/SAM/SOM, 경쟁사, 기회/리스크)
- 전략 검증 결과 (Go 판정, 리스크 대응)
- 비즈니스 모델 (`--bm` 시에만: 수익 모델, 가격 티어, 유닛 이코노믹스, BM Score)
- CEO 최종 방향 (승인 사유, 조정 사항)

## 독립 실행 vs 파이프라인 모드

| 모드 | 차이 |
|------|------|
| **독립 실행** | 사용자가 직접 호출. #1~#9 완료 후 결과를 사용자에게 보고하고 종료. |
| **파이프라인 모드** | auto-dev가 호출. #1~#9 완료 후 idea-brief.md + 전체 산출물을 auto-dev에 반환. auto-dev가 Design Phase(#10~)로 진행. |

## 경계

- 이 스킬은 Idea Phase **오케스트레이션만** 수행한다
- 트렌드 스코어링은 researcher 에이전트(Trend Scoring 모드), BM 설계는 bm-designer 에이전트, 시장 조사는 researcher 에이전트, 전략 검증은 CSO 에이전트가 각각 담당
- 에이전트를 직접 호출하지 않는다 — 메인 모델에게 에이전트 호출을 요청하고 결과를 받아 다음 단계로 전달
- Design Phase 이후 단계는 design-loop, build-loop, verify-loop 스킬이 담당
