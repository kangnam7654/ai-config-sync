# Response: 코드 스니펫 CLI 도구 프로젝트

## 요청 분석

사용자가 새로운 서비스(코드 스니펫 저장/검색 CLI 도구)를 만들고 싶다고 요청했습니다. 이 요청에는 구현이 필요하므로, Design-First Development 원칙에 따라 설계문서를 먼저 작성해야 합니다.

## 실제 응답 내용

좋은 아이디어네요! 개발자용 코드 스니펫 CLI 도구를 만들어 봅시다.

구현에 들어가기 전에, Design-First 원칙에 따라 설계문서를 먼저 작성해야 합니다. 몇 가지 요구사항을 확인하고 싶습니다:

### 확인이 필요한 사항

1. **언어/런타임**: 어떤 언어로 만들고 싶으신가요? (Python CLI, Go, Rust, Node.js 등)
2. **저장소**: 스니펫을 어디에 저장할 건가요?
   - 로컬 파일 (SQLite, JSON, YAML 등)
   - 원격 서버 (API 서버 + DB)
   - Git 기반 (이 프로젝트처럼 git repo로 동기화)
3. **팀 공유 방식**:
   - Git repo 공유 (간단, 오프라인 가능)
   - 중앙 서버 API (실시간, 권한 관리 가능)
   - 둘 다 지원
4. **핵심 기능 우선순위**: 아래 중 MVP에 꼭 들어가야 할 것은?
   - 스니펫 저장 (코드 + 설명 + 태그)
   - 검색 (태그, 언어, 키워드)
   - 팀/그룹 공유
   - 즐겨찾기 / 사용 빈도 추적
   - 클립보드 복사
   - 에디터 통합 (VS Code 등)
5. **인증**: 팀 공유 시 인증은 어떤 방식으로? (GitHub OAuth, API key, 없음 등)

### 설계문서 작성 계획

요구사항이 확정되면, 아래 구조로 설계문서를 작성하겠습니다:

```
{project}/docs/snippet-cli/
  design.md          # 설계문서 (목적, 아키텍처, 데이터 흐름, API, 파일 구조, 의사결정)
  architecture.mmd   # 시스템 아키텍처 다이어그램
  data-flow.mmd      # 데이터 흐름 다이어그램
```

설계문서에는 다음 필수 섹션이 포함됩니다:
- **목적**: 해결하려는 문제 + 성공 기준
- **아키텍처**: Mermaid 다이어그램으로 컴포넌트 간 관계
- **데이터 흐름**: 입력 -> 처리 -> 출력 경로
- **API 설계**: CLI 명령어 인터페이스 + 내부 함수 시그니처
- **파일 구조**: 생성할 파일 전체 경로
- **의사결정 근거**: 채택/기각한 대안과 이유

위 질문들에 대해 알려주시면 바로 설계문서 작성을 시작하겠습니다!

## 프로세스 요약 (이 응답이 따르는 규칙)

1. **Session Start Protocol**: `git status` -> `git pull --rebase origin main` 실행 완료
2. **Design-First Development**: 구현 코드 작성 전 설계문서 작성 + 사용자 승인 필요
3. **NEVER 규칙 #5 준수**: 설계문서 완료 + 사용자 승인 이전에 구현 코드를 작성하지 않음
4. **요구사항 확인**: 모호한 부분(언어, 저장소, 공유 방식 등)을 사용자에게 질문
5. **Writer -> Critic 루프**: 설계문서 작성 시 doc-writer -> doc-critic 루프를 거칠 예정
