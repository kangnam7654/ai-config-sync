# Doc Loop

Writer와 doc-critic 에이전트를 교대로 호출하여 문서를 반복 개선하는 오케스트레이션 워크플로우.

## 왜 루프가 필요한가

문서는 한 번에 완벽하게 나오지 않는다. Writer가 초안을 작성하면 doc-critic이 모드별 가중 기준으로 채점하고, 가장 약한 부분에 대해 구체적 피드백을 준다. 이 피드백을 반영하여 문서를 개선하는 루프를 반복하면, 독자(사람 또는 LLM)에게 명확하고 실행 가능한 문서가 만들어진다.

## 모드 분기

이 스킬은 문서의 대상 독자에 따라 두 가지 경로로 분기한다:

| 대상 독자 | Writer 에이전트 | doc-critic 모드 | 예시 |
|-----------|----------------|----------------|------|
| 사람 | `doc-writer-human` | HUMAN | README, 설계문서, API 문서, 온보딩 가이드, 변경로그 |
| LLM | `doc-writer-llm` | LLM | CLAUDE.md, 에이전트 정의(.md), 스킬 SKILL.md, 시스템 프롬프트 |

대상 독자가 불명확하면 사용자에게 "이 문서의 주요 독자는 사람인가요, LLM인가요?" 라고 확인한다.

## 이 스킬을 사용하지 않는 경우

- 실행 계획 요청 (예: "계획 세워줘", "리팩터링 플랜") → `plan-loop` 스킬 사용
- 단일 코드 작업 (예: "이 함수 고쳐줘", "테스트 추가해줘") → 직접 수행
- 단순 질문 (예: "이 코드 뭐 하는 거야?") → 직접 답변

## 워크플로우

### 1단계: Writer 호출

모드에 따라 적절한 Writer 에이전트를 호출하여 문서 초안을 받는다.

- **사람용**: `doc-writer-human` 에이전트(subagent_type: `doc-writer-human`) 호출
- **LLM용**: `doc-writer-llm` 에이전트(subagent_type: `doc-writer-llm`) 호출
- 사용자의 요청 내용, 문서 목적, 대상 독자 정보를 Writer에게 전달

### 2단계: 초안 검증 게이트

doc-critic에게 보내기 전에 최소 품질을 확인한다. 명백히 불충분한 출력으로 critic의 리소스를 낭비하지 않기 위한 사전 필터다.

**즉시 REJECT 조건** (하나라도 해당하면):
- 문서가 비어 있음
- 본문이 3문장 이하

REJECT 시: Writer에게 아래 메시지와 함께 재작성을 요청하고 1단계로 복귀한다.
> "출력이 불충분합니다. 최소 요구사항을 충족하는 초안을 작성하세요."

검증 게이트를 통과하면 3단계로 진행한다.

### 3단계: doc-critic 호출

doc-critic 에이전트(subagent_type: `doc-critic`)를 호출한다. 1단계에서 선택한 모드를 doc-critic에게 전달한다.

**HUMAN 모드 채점 기준:**
| 기준 | 가중치 |
|------|--------|
| Readability | 30% |
| Structure | 25% |
| Examples | 20% |
| Completeness | 15% |
| Accuracy | 10% |

**LLM 모드 채점 기준:**
| 기준 | 가중치 |
|------|--------|
| Precision | 30% |
| Executability | 25% |
| Completeness | 20% |
| Conciseness | 15% |
| Error handling | 10% |

- **PASS 기준**: 총점 > 8.00 AND 최고 가중치 기준(Readability 또는 Precision) >= 8
- 라운드당 정확히 1개의 피드백 — 가장 낮은 점수 기준의 가장 큰 이슈를 지적

### 4단계: 결과 처리

- **REJECT**: critic의 피드백을 Writer에게 전달하여 수정 요청 → 2단계로 복귀
- **PASS**: 최종 문서를 사용자에게 전달하고 루프 종료

## 라운드 보고

매 라운드마다 사용자에게 아래 형식으로 진행 상황을 보고한다:

```
[라운드 N] 점수: X.XX | 결과: PASS/REJECT | 피드백: (1줄 요약)
```

## NEVER 규칙

1. NEVER: 문서가 아닌 산출물(실행 계획, 코드, 답변)을 이 루프로 처리하지 마라. 실행 계획은 `plan-loop` 스킬을, 코드와 답변은 직접 수행을 사용한다.
2. NEVER: 초안 검증 게이트(2단계)를 건너뛰고 doc-critic을 호출하지 마라. 모든 Writer 출력은 반드시 게이트를 통과해야 한다.
3. NEVER: Writer 에이전트가 doc-critic을 직접 호출하도록 허용하지 마라. critic 호출은 메인 모델만 수행한다.
4. NEVER: 사람용 문서에 LLM 모드를, LLM용 문서에 HUMAN 모드를 적용하지 마라. 모드 불일치는 채점 기준이 맞지 않아 무의미한 피드백을 생성한다.

## 제한 규칙

- **최대 반복 횟수: 5회.** 검증 게이트 REJECT도 횟수에 포함한다. 5회 연속 REJECT 시 현재까지 가장 높은 점수를 받은 문서와 미해결 이슈를 사용자에게 보고하고 판단을 요청한다.

## 엣지 케이스

| 상황 | 처리 |
|------|------|
| Writer가 에러를 반환하거나 응답 없음 | 1회 재시도한다. 재시도도 실패하면 사용자에게 "Writer 호출에 실패했습니다. 요청을 다시 시도하시겠습니까?" 라고 보고하고 루프를 종료한다. |
| doc-critic이 에러를 반환하거나 응답 없음 | 1회 재시도한다. 재시도도 실패하면 현재 문서를 사용자에게 전달하며 "doc-critic 채점 불가. 수동 검토가 필요합니다." 라고 안내한다. |
| 대상 독자가 불명확 (사람/LLM 구분 불가) | Writer를 호출하지 않는다. 사용자에게 "이 문서의 주요 독자는 사람인가요, LLM인가요?" 라고 되묻는다. |
| Writer가 이전 라운드와 동일한 문서를 반환 (critic 피드백 미반영) | 해당 라운드를 REJECT로 처리한다. critic 피드백 원문을 재인용하여 "이전 피드백이 반영되지 않았습니다. 아래 피드백을 반영하여 수정하세요:" 메시지와 함께 Writer에게 재전달한다. |
| critic이 PASS를 반환했으나 총점이 8.00 이하 (채점 오류 의심) | PASS를 무시하고 REJECT로 처리한다. critic에게 재채점을 요청한다. |
| 5라운드 소진 시 최선 문서 선택 | 5라운드 중 가장 높은 총점을 받은 문서를 사용자에게 전달한다. 모든 라운드의 점수를 함께 보고한다. |
| 사용자가 루프 도중 중단을 요청 | 즉시 루프를 중단한다. 현재까지의 최선 문서를 사용자에게 전달한다. |
