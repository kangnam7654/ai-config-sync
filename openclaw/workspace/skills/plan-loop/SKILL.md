# Plan Loop

planner와 plan-critic 에이전트를 교대로 호출하여 실행 계획을 반복 개선하는 오케스트레이션 워크플로우.

## 왜 루프가 필요한가

플랜은 한 번에 완벽하게 나오지 않는다. planner가 초안을 작성하면 plan-critic이 Clarity, Completeness, Feasibility, Dependencies, Risk, Scope 6가지 기준으로 채점하고, 가장 약한 부분에 대해 구체적 피드백을 준다. 이 피드백을 반영하여 플랜을 개선하는 루프를 반복하면, 실행 가능성이 높고 빈틈 없는 플랜이 만들어진다.

## 이 스킬을 사용하지 않는 경우

- 단일 작업 요청 (예: "이 함수 이름 바꿔줘", "버그 고쳐줘") → 플랜 없이 직접 수행
- 문서 산출물 요청 (예: "README 작성해줘", "API 문서 만들어줘") → Writer → Critic 루프 사용
- 단순 질문 (예: "이 코드 뭐 하는 거야?", "Go와 Rust 차이가 뭐야?") → 직접 답변

## 워크플로우

### 1단계: planner 호출

planner 에이전트(subagent_type: `planner`)를 호출하여 플랜 초안을 받는다.

- 사용자의 요청 내용과 프로젝트 컨텍스트를 planner에게 전달
- planner가 단계별 실행 계획을 작성하여 반환

### 2단계: 초안 검증 게이트

plan-critic에게 보내기 전에 최소 품질을 확인한다. 명백히 불충분한 출력으로 critic의 리소스를 낭비하지 않기 위한 사전 필터다.

**즉시 REJECT 조건** (하나라도 해당하면):
- planner 출력이 비어 있음
- 실행 단계가 2개 이하

REJECT 시: planner에게 아래 메시지와 함께 재작성을 요청하고 1단계로 복귀한다.
> "출력이 불충분합니다. 최소 3단계 이상의 구체적 플랜을 작성하세요."

검증 게이트를 통과하면 3단계로 진행한다.

### 3단계: plan-critic 호출

plan-critic 에이전트(subagent_type: `plan-critic`)를 호출한다.

plan-critic의 채점 기준:
| 기준 | 가중치 |
|------|--------|
| Clarity | 30% |
| Completeness | 20% |
| Feasibility | 15% |
| Dependencies | 15% |
| Risk | 10% |
| Scope | 10% |

- **PASS 기준**: 총점 > 8.00 AND Clarity >= 8
- 라운드당 정확히 1개의 피드백 — 가장 낮은 점수 기준의 가장 큰 이슈를 지적

### 4단계: 결과 처리

- **REJECT**: critic의 피드백을 planner에게 전달하여 수정 요청 → 2단계로 복귀
- **PASS**: 최종 플랜을 사용자에게 전달하고 루프 종료

## 라운드 보고

매 라운드마다 사용자에게 아래 형식으로 진행 상황을 보고한다:

```
[라운드 N] 점수: X.XX | 결과: PASS/REJECT | 피드백: (1줄 요약)
```

## NEVER 규칙

1. NEVER: 단계별 실행 계획이 아닌 산출물(문서, 코드, 답변)을 이 루프로 처리하지 마라. 문서는 Writer → Critic 루프를, 코드와 답변은 직접 수행을 사용한다.
2. NEVER: 초안 검증 게이트(2단계)를 건너뛰고 plan-critic을 호출하지 마라. 모든 planner 출력은 반드시 게이트를 통과해야 한다.
3. NEVER: planner 에이전트가 plan-critic을 직접 호출하도록 허용하지 마라. critic 호출은 메인 모델만 수행한다.
4. NEVER: plan-critic의 PASS/REJECT 판정을 메인 모델이 임의로 뒤집지 마라. 단, 총점이 8.00 이하인데 PASS를 반환한 경우(채점 오류)는 REJECT로 처리한다.

## 제한 규칙

- **최대 반복 횟수: 5회.** 검증 게이트 REJECT도 횟수에 포함한다. 5회 연속 REJECT 시 현재까지 가장 높은 점수를 받은 플랜과 미해결 이슈를 사용자에게 보고하고 판단을 요청한다.

## 엣지 케이스

| 상황 | 처리 |
|------|------|
| planner가 에러를 반환하거나 응답 없음 | 1회 재시도한다. 재시도도 실패하면 사용자에게 "planner 호출에 실패했습니다. 요청을 다시 시도하시겠습니까?" 라고 보고하고 루프를 종료한다. |
| plan-critic이 에러를 반환하거나 응답 없음 | 1회 재시도한다. 재시도도 실패하면 현재 플랜을 사용자에게 전달하며 "plan-critic 채점 불가. 수동 검토가 필요합니다." 라고 안내한다. |
| 사용자 요청이 너무 모호하여 플랜 수립 불가 (예: "뭔가 해줘", "개선해줘") | planner를 호출하지 않는다. 사용자에게 "플랜을 수립하려면 구체적인 목표가 필요합니다. 어떤 결과물을 원하시나요?" 라고 되묻는다. |
| planner가 이전 라운드와 동일한 플랜을 반환 (critic 피드백 미반영) | 해당 라운드를 REJECT로 처리한다. critic 피드백 원문을 재인용하여 "이전 피드백이 반영되지 않았습니다. 아래 피드백을 반영하여 수정하세요:" 메시지와 함께 planner에게 재전달한다. |
| critic이 PASS를 반환했으나 총점이 8.00 이하 (채점 오류 의심) | PASS를 무시하고 REJECT로 처리한다. critic에게 재채점을 요청한다. |
| 5라운드 소진 시 최선 플랜 선택 | 5라운드 중 가장 높은 총점을 받은 플랜을 사용자에게 전달한다. 모든 라운드의 점수를 함께 보고한다. |
| 사용자가 루프 도중 중단을 요청 | 즉시 루프를 중단한다. 현재까지의 최선 플랜을 사용자에게 전달한다. |
